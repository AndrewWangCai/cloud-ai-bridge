# 协作房间：多人 + 多 AI 共建一个项目（v1）

把你的项目变成一个「房间」，邀请别人进来，多人一起用各自的 AI（Claude/Codex/DeepSeek）协作开发。

## 怎么用
**房主（项目所有者）**：
1. 【配置】→「协作房间」→ 输入对方用户名 →【邀请】。
2. 正常【激活并连接沙箱】。被邀请的人即可加入。

**协作者（被邀请的人）**：
1. 登录后，项目下拉里会出现「👥 房主/项目（协作）」。
2. 选它 →【激活并连接】（你不会重启房主的沙箱，只是加入）。

进入后：
- **共享同一个沙箱/工作区/终端/预览**：谁的操作大家都能看到（输出实时广播）。
- **聊天区**顶部出现协作条：在线成员、当前文件锁、锁/解锁输入框。
- 成员发言、加入/离开都会实时显示。

## 文件互斥锁（避免互相覆盖）
担心两个人的 AI 同时改同一个文件？用锁：
1. 在协作条的输入框填要改的文件（逗号分隔，如 `src/app.js,index.html`）→【🔒锁】。
2. 这些文件就被你锁住了；**别人再想锁其中任何一个会被拒绝**（提示被谁锁了）。
3. 锁定信息会写进项目的 `LOCKS.md` 和 `CLAUDE.md`/`AGENTS.md`，**所有人的 AI 都会被告知「只有当前操作者是锁持有人时才能改，否则不要改」**。
4. 改完点【解锁】（不填=解我全部）。

> 你描述的场景正是这样：A 锁了 ABC，B 想改 BCD → B 只能锁到 D，BC 提示被 A 占用；B 的 AI 读到锁信息后只改 D、不动 BC。

### 锁是「建议性 + 声明性」的（诚实说明）
AI 在容器里直接写文件，平台没法逐次拦截磁盘写入。所以锁的保证是两层：
- **硬保证**：同一文件不会被两个人**同时持有**（claim 冲突直接拒绝）。
- **软保证**：锁信息注入到 AI 的上下文（LOCKS.md / CLAUDE.md / AGENTS.md），AI 被明确要求尊重锁持有人——它会遵守，但前提是 AI 听话。
（未来可加「OS 级只读」硬锁作为可选增强。）

## 权限与边界（v1）
- 只有**房主**能邀请/移除协作者、激活沙箱、绑定 Key。
- 协作者能：连接、看终端/预览、发言、锁/解锁、用自己的指令驱动房主沙箱里的 AI。
- v1 使用**房主项目沙箱里已绑定的 Key / 会员登录态**。如果要按协作者各自 Key 计费，需要后续做 per-member agent 或凭证切换。
- 房主可随时移除协作者；移除后对方无法再进入。

## 已知限制（v1，后续迭代）
- 共享一个终端：同时多人输入会交错（建议口头约定「谁在敲」，或用锁+聊天协调）。
- 锁是建议性（见上）。
- 移除协作者会释放该协作者持有的锁；普通断线不会自动释放锁，避免网络抖动导致锁丢失。
- 还没有分支/并发执行、细粒度角色（旁观/只读）——放到 v2。
